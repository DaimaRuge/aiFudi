# Horizon OS 统一整合方案（基于新增功能定义表重构）

> 本文档不是把新增功能"塞进原框架"，而是基于大屏 + 语音 + Agent + 服务生态方向，对整个 Horizon OS 做一次产品级重构。
> 从 **系统架构、信息架构、UI 架构、UE 策略、功能组织逻辑、Agent 主线、控制与服务关系** 几个层面统一梳理。

---

## 前置说明

1. **研发清单**目前看是控制能力定义表，重点覆盖灯光、电器、能源、空调、水路、设备等房车核心控制能力。
2. 部分功能项（如"冬季通风口控制"）文本上可能尚未完整结束；本方案先基于已给出的完整内容整合，缺失项后续可继续并入。

## 一、核心结论：Horizon OS 不能再按"控制面板"思路组织

### 1.1 新系统的本质定义

Horizon OS 不是房车控制屏的放大版，而是：

> **以 Agent 为核心、语音为主入口、空间控制为基础、服务生态为延展的大屏房车智能空间操作系统。**

它必须同时覆盖两类能力：

**A. 控制类能力**（用于真正管理房车空间）

- 灯光
- 空调与通风
- 水路与热管理
- 能源与供配电
- 设备与机构件
- 电器与影音
- 底盘与宿营动作

**B. 服务类能力**（用于放大"大屏 + AI 主机"的价值）

- 音乐
- 视频
- 地图
- 天气
- 日历
- 行程与旅行建议
- 通知、提醒、主动服务
- 未来可扩展的生态服务

---

### 1.2 产品组织的根本原则

如果继续按传统方式做，很容易变成"大而杂控制墙"。Horizon OS 必须遵循以下 **5 个原则**：

#### 原则 1：用户面对的是"房车管家"，不是一堆设备按钮

用户首先应该说：
- "打开观影模式"
- "看看今天适合去哪露营"
- "把客厅灯暖一点"
- "准备洗澡"
- "现在还能用多久电"

而不是先去找：灯光页、水路页、DCDC 页、设备页、电器页。

#### 原则 2：场景 / 空间 / 系统 三层并存，但层级不同

为避免系统变成杂乱面板，必须同时支持三种组织方式：

| 层级 | 组织方式 | 适用场景 | 示例 |
|:---:|:---:|:---:|:---|
| 第一层 | 按场景 | 最符合用户任务 | 一键宿营、观影、做饭、洗浴、睡眠、离车、晨起、夜间 |
| 第二层 | 按空间 | 最符合房车生活理解 | 客厅、卧室、卫浴、吧台/厨房、室外 |
| 第三层 | 按系统 | 最适合深度控制 | 灯光、环境、水路、能源、设备、电器、底盘/宿营机构 |

**结论：用户默认走"场景/空间"，系统深控才进入"系统层"。**

#### 原则 3：Agent 必须成为主线，不是附加功能

Agent 不是页面角落里的"麦克风按钮"，而是 Horizon OS 的核心结构：
- 承接自然语言
- 理解任务
- 调度系统
- 主动提醒
- 生成建议
- 把复杂系统翻译成用户能理解的表达

#### 原则 4：控制与服务要并行，不要互相挤压

Horizon OS 既要能控制房车，也要承载旅行生活方式。系统架构必须明确分为两条主轴：

- **空间控制主轴**：环境、能源、水路、设备、电器、底盘
- **数字服务主轴**：地图、音乐、视频、天气、日历、旅行建议

二者通过以下三个中枢打通：**房车管家**、**首页**、**消息中心/场景引擎**

#### 原则 5：大屏必须发挥"大屏价值"，而不是只显示状态卡

形态是 **43–75 英寸 16:9 电视**，必须承担：
- 更强的信息总览
- 更强的空间感与氛围感
- 更强的路线/地图可视化
- 更强的音乐/视频体验
- 更自然的 Agent 对话反馈
- 更舒展的场景卡片与任务流程

## 二、Horizon OS 总体系统架构

### 2.1 五层架构

```
┌─────────────────────────────────────────────────────────┐
│                    L1 交互入口层                         │
│  房车管家(语音) │ 大屏首页 │ 蓝牙遥控器 │ 手机APP/PAD │ 硬开关 │
├─────────────────────────────────────────────────────────┤
│                    L2 智能中枢层                         │
│  Agent调度 │ 语音理解(ASR/NLU) │ 消息中枢 │ 场景引擎     │
│  安全策略引擎 │ 状态总线/设备数字孪生                     │
├─────────────────────────────────────────────────────────┤
│                    L3 业务能力层                         │
│  场景/空间控制 │ 环境/照明 │ 水路/热管理 │ 能源/供配电    │
│  设备/机构件 │ 电器/影音 │ 地图 │ 音乐 │ 视频 │ 天气/日历  │
├─────────────────────────────────────────────────────────┤
│                    L4 接入控制层                         │
│  IoT协议网关 │ 智能配电箱 │ 空调接口 │ 水路泵阀接口      │
│  BMS/充放电接口 │ 机构控制接口 │ 音视频接口 │ 传感器接口   │
├─────────────────────────────────────────────────────────┤
│                    L5 运行平台层                         │
│  AI主机 │ 本地/云模型协同 │ 语音引擎 │ 日志/配置/OTA     │
│  权限 │ 电视大屏(显示输出)                                │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Horizon OS 的真正"骨架"

**不是：** 一个首页 + 六七个控制分页 + 几十个设备开关卡片

**而是：**
- 一个房车管家主线
- 一套场景引擎
- 一套空间结构
- 一组系统深控能力
- 一条服务生态扩展轴

## 三、统一功能体系：把新增控制清单重新归类

研发清单本身是按"灯光/电器/能源/空调/水路/设备"列的。这个分类适合研发接口定义，但不适合直接拿来做产品 IA。产品层应重构为统一体系。

### 3.1 Horizon OS 功能总树

```
Horizon OS
├── 1. Agent 与全局能力
│   ├── 房车管家
│   ├── 语音交互
│   ├── 消息中心
│   ├── 场景引擎
│   ├── 主动提醒
│   ├── 建议与服务分发
│   └── 控制中心
│
├── 2. 空间控制体系
│   ├── 客厅空间
│   ├── 卧室空间
│   ├── 卫浴空间
│   ├── 厨房/吧台空间
│   └── 室外空间
│
├── 3. 系统控制体系
│   ├── 3.1 灯光系统
│   ├── 3.2 环境系统（空调 + 通风）
│   ├── 3.3 水路与热管理系统
│   ├── 3.4 能源系统
│   ├── 3.5 设备与机构系统
│   ├── 3.6 电器与影音系统
│   └── 3.7 底盘/宿营动作系统
│
├── 4. 数字服务体系
│   ├── 地图导航
│   ├── 音乐
│   ├── 视频
│   ├── 天气
│   ├── 日历
│   ├── 行程与旅行建议
│   └── 生态服务扩展
│
└── 5. 系统服务与安全体系
    ├── 故障告警
    ├── 设备状态历史
    ├── 过滤寿命/保养提醒
    ├── 自动联动策略

    └── 多端状态同步
```

### 3.2 六大控制域重构

#### A. 灯光域

| 空间 | 灯具类型 | 控制能力 |
|:---:|:---:|:---:|
| 客厅 | 照明灯、氛围灯、辅助灯（筒灯/轨道灯） | 开关、亮度、色温、场景联动 |
| 卧室 | 照明灯、氛围灯 | 开关、亮度、色温 |
| 卫浴 | 照明灯 | 开关、亮度 |

**交互入口：** 语音、大屏、APP/PAD、硬开关

**产品层处理方式：** 不要直接展示"每一盏灯的按钮矩阵"，而应采用：
- **空间模式：** 客厅/卧室/卫浴
- **场景模式：** 晨起/观影/睡眠/露营/起夜
- **系统深控：** 灯具级别单独调节

#### B. 环境域（空调 + 通风统一）

**空调热管理控制能力：**

| 控制项 | 详细参数 |
|:---:|:---|
| 电源 | 开关机 |
| 模式 | 自动、制冷、制热、新风、强力、节能、睡眠 |
| 温度单位 | ℃ / ℉ |
| 设定温度 | 16~30℃ / 60~86℉ |
| 风量 | 高/中/低 |
| 定时 | 1h~8h |
| 故障反馈 | 故障代码 |

**通风系统控制能力：**

| 设备 | 控制能力 |
|:---:|:---|
| 客厅换气扇 | 开关、风速 1~10 档 |
| 卫生间换气扇 | 开关、风速 1~10 档 |
| 冬季通风口 | 开关、风向（进风/排风）、风速 |

**产品层处理方式：** 环境系统应让用户感知的是"舒适/温度/空气/睡眠/新风"，而不是先区分空调、换气扇、冬季通风口。

#### C. 水路与热管理域

| 子系统 | 控制能力 |
|:---:|:---|
| **水箱状态** | 清水箱水量、灰水箱水量、清水箱温度、灰水箱温度 |
| **水泵系统** | 生活水泵、自吸水泵 |
| **排水系统** | 电动排水阀 |
| **净水循环** | 清水箱杀菌、水路杀菌、RO 反渗透净水器（TDS、滤芯寿命） |
| **热水系统** | 热水器电源、预约加热、加热时长 |
| **防冻系统** | 水路防冻强制运行、洗澡预约加热、目标温度、加热时长 |

**产品层处理方式：** 对用户应围绕三个任务组织——**用水、洗澡、净水与安全、排水与防冻**。

#### D. 能源域

| 子系统 | 控制能力 |
|:---:|:---|
| **充电** | 慢充、快充、太阳能（电流、电压、预计充满时间） |
| **放电** | V2L/V2H/V2V（放电状态、本次放电量、剩余最低电量阈值） |
| **DCAC** | 开/关 |
| **MPPT** | 开/关 |
| **DCDC** | 开/关，12V 电池低于 20% 自动开启 |
| **电池与供电** | 动力电池、12V 电池、供电模式、能耗趋势 |

**产品层处理方式：** 能源不能只做状态页，必须兼顾总览、趋势、安全阈值、自动策略、故障/异常、场景联动。例如宿营模式自动评估电量是否足够、夜间节能模式自动调整部分负载、12V 低电量自动唤醒 DCDC 并通知。

#### E. 设备与机构域（升级为"宿营动作系统"）

| 设备 | 控制能力 |
|:---:|:---|
| **遮阳棚** | 电源开关、风感/雨感自动收回、与一键宿营联控、状态显示 |
| **踏步** | 与乘客门联动、与一键宿营联动、状态显示 |
| **支撑腿** | 开关、一键宿营联动、场景自调整、报警联动、状态显示 |
| **遮阳帘** | 开关、场景定义、状态显示 |
| **升降桌** | 升降、位置记忆、状态显示 |

**产品层处理方式：** 不应只是"设备页"，而应形成更高价值的"宿营与空间机构系统"，与场景引擎深度关联（一键宿营、一键离营、用餐模式、观影模式、户外休闲模式）。

#### F. 电器与影音域

**生活电器：** 冰箱（上下电/开关/温度调节）、洗衣机（上下电/工作控制）、油烟机（上下电/开始工作）、电磁炉（上下电）、电烤箱（上下电）

**影音设备：** 电视机、投影仪、幕布、功放

**产品层处理方式：** 影音设备不要藏在控制页里，而要和大屏娱乐体系打通（观影模式、投影链路、声场控制、视频服务入口）。

## 四、Horizon OS 的信息架构重构

### 4.1 不建议直接把导航做成"灯光/电器/水路/能源/空调/设备"一排

这会非常像传统控制系统。对于大屏 + 语音 + Agent 来说，主导航应该更符合用户心智，而不是研发模块心智。

### 4.2 推荐 IA：主导航 5+2 模型

```
┌──────────────────────────────────────────────────┐
│  L0 全局常驻层                                    │
│  房车管家 │ 全局语音状态 │ 消息提醒 │ 控制中心入口  │
├──────────────────────────────────────────────────┤
│  L1 主导航                                        │
│  首页 │ 空间 │ 场景 │ 出行 │ 娱乐 │ 服务 │ 控制中心  │
├──────────────────────────────────────────────────┤
│  L2 二级结构                                      │
│                                                  │
│  首页: 今日状态│管家建议│高频场景│旅行灵感│快捷入口  │
│  空间: 客厅│卧室│卫浴│厨房/吧台│室外│系统深控入口  │
│  场景: 宿营│离营│观影│做饭│洗澡│睡眠│晨起│自定义   │
│  出行: 地图│天气│日历│路线│营地/景点│旅行建议       │
│  娱乐: 音乐│视频│投屏│影音设备│声场与氛围           │
│  服务: 消息中心│告警│用水/用电提醒│管家建议│保养     │
│                                                  │
├──────────────────────────────────────────────────┤
│  L3 系统深控                                      │
│  灯光│环境│水路与热管理│能源│设备与机构              │
│  电器与影音│底盘/宿营动作                           │
└──────────────────────────────────────────────────┘
```

### 4.3 这个 IA 的好处

- **对用户：** 用户先找的是"我现在要做什么""我在哪个空间""管家建议我什么"
- **对系统：** 系统深控仍完整保留，不丢控制能力
- **对 Agent：** Agent 可以天然穿透多个系统，不会被单模块割裂

## 五、UI 架构：大屏不应该是"按钮墙"

### 5.1 大屏 UI 的正确结构

推荐采用 **"舞台式首页 + 卡片式分发 + 全屏深控页 + 浮层控制中心"**。

### 5.2 UI 分层

| 层级 | 名称 | 内容 |
|:---:|:---:|:---|
| Layer 1 | 背景氛围层 | 房车旅行主题视觉（晨光/夕阳/湖/海/公路/国家公园），动态天气与时间氛围 |
| Layer 2 | Agent 主舞台层 | 房车管家形象/对话区，语音状态反馈，主动消息与建议 |
| Layer 3 | 业务卡片层 | 状态总览卡、场景卡、空间卡、地图/音乐/视频推荐卡、告警/服务卡 |
| Layer 4 | 全屏模块层 | 空间页、场景页、出行页、娱乐页、系统深控页 |
| Layer 5 | 浮层与控制中心 | 音量、亮度、网络、投屏、连接状态、摄像头、语言 |

### 5.3 UI 呈现原则

| 页面 | 不应该是 | 应该是 |
|:---:|:---:|:---|
| **首页** | 状态墙 | 管家主舞台 + 状态摘要 + 高频场景 + 内容分发 + 服务建议 |
| **空间页** | 系统树 | 每个空间现在怎么样 + 能快速做什么 |
| **系统深控页** | — | 灯具级控制、温度阈值、DCDC/MPPT、TDS/滤芯寿命、故障详情、定时设置、场景参数 |

## 六、UE 策略：如何让 Agent 和语音成为主线

### 6.1 主交互不是"点控"，而是"任务驱动"

Horizon OS 的 UE 主线应从"功能点击"改成"任务执行"。

### 6.2 三种任务入口

**入口一：用户主动说**
- "打开宿营模式"
- "把卧室灯调暗一点"
- "检查一下清水还剩多少"
- "准备洗澡"
- "播放适合晚霞的音乐"

**入口二：用户在首页/场景页选卡片**
- 宿营、观影、睡眠、去最近营地、洗澡准备

**入口三：Agent 主动建议**
- "外面开始下雨，是否收回遮阳棚？"
- "12V 电池已低于 20%，已自动开启 DCDC"
- "灰水箱接近满载，建议排水"
- "今晚气温较低，是否开启防冻保护？"
- "距离目的地还有 40 分钟，是否播放旅行歌单？"

### 6.3 Agent 在 UE 中的四个职责

| 职责 | 说明 | 示例 |
|:---:|:---|:---|
| **翻译** | 把复杂控制翻译成自然语言任务 | — |
| **编排** | 把用户一句话拆成多个动作 | "准备观影" → 放下幕布 + 打开投影 + 打开功放 + 调整灯光 |
| **解释** | 告诉用户当前发生了什么 | "空调已切到睡眠模式，温度设为 24℃" |
| **主动** | 基于状态和情境发起建议与告警 | "清水箱当前剩余 38%，预计还能支持一天正常使用" |

## 七、控制与服务如何共存

### 7.1 两条主轴

| 控制主轴 | 服务主轴 |
|:---:|:---|
| 灯光 | 地图 |
| 环境 | 天气 |
| 水路 | 日历 |
| 能源 | 音乐 |
| 设备 | 视频 |
| 电器 | 旅行建议 |
| 底盘/宿营 | 生态服务 |

### 7.2 三个连接点

| 连接点 | 说明 |
|:---:|:---|
| **首页** | 同时承载当前状态、场景入口、内容入口、服务入口 |
| **场景引擎** | 同一场景既涉及控制也涉及服务（如"宿营夜晚"：氛围灯 + 空调睡眠 + 轻音乐 + 天气与次日行程提醒） |
| **消息中心** | 既展示设备告警/滤芯寿命/故障代码，也展示旅行建议/天气提醒/行程提醒/推荐内容 |

## 八、关键产品能力：必须新增"场景引擎"

研发清单里已出现多个联动线索（遮阳棚与一键宿营联动、踏步与车门/宿营联动、支撑腿与宿营/报警联动、DCDC 自动策略、风感/雨感自动收回、故障提醒与状态显示），说明必须建设场景引擎。

### 8.1 推荐场景体系

| 场景类型 | 场景列表 |
|:---:|:---|
| **基础场景** | 一键宿营、一键离营、睡眠、晨起、夜间 |
| **生活场景** | 观影、做饭、洗澡、阅读、户外休闲 |
| **安全场景** | 低电量保护、雨天自动收棚、低温防冻、灰水箱接近满自动提醒、热管理故障提醒 |

### 8.2 场景引擎在系统中的价值

场景引擎是连接 **Agent、控制能力、自动化、服务内容、用户习惯** 的关键中间层。

## 十一、对这个项目最重要的判断

### 11.1 真正该守住的不是"功能全"，而是"主线清晰"

如果只是追求全覆盖，最后一定会变成传统大控制屏。Horizon OS 必须同时守住两件事：

1. **控制能力要全：** 灯光、空调、水路、能源、设备、电器、宿营，都必须能控、能看、能联动。
2. **用户路径要轻：** 用户默认不应先钻系统树，而应——说一句、选一个场景、进一个空间、看管家建议。

### 11.2 Horizon OS 的最佳定位

> **Horizon OS 是一套以房车管家为核心入口、以场景与空间为主要组织方式、以系统深控为专业支撑、以地图音乐视频等服务生态为延展的大屏房车智能空间操作系统。**
